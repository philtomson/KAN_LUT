# Implementation Plan: FPGA KAN-LUT Hardware Accelerators

This plan details the implementation of the two hardware paradigms for Kolmogorov-Arnold Network (KAN) Look-Up Table (LUT) inference:

1. **Paradigm 1 (Fully-Unrolled Custom RTL)**: A Julia script generating custom, pipelined, single-cycle-throughput SystemVerilog modules and ROM initializations specifically for the MNIST network (`196 -> 32 -> 10`).
2. **Paradigm 2 (Parameterized Generic IP Core)**: A reusable, multi-cycle SystemVerilog design at the top-level `hardware/` directory that reads inputs and processes layer activations sequentially using parameterized execution units and dual-port BRAMs.

We will create a custom RTL generation script under a new directory: `examples/MNIST/FPGA/`.

#### [generate_rtl.jl](file:///home/phil/devel/FPGA/KAN_LUT/examples/MNIST/FPGA/generate_rtl.jl)

* Writes SystemVerilog files:

  * `mnist_kan_top.sv`: Connects Layer 1 and Layer 2 with an inter-layer pipeline register boundary.
  * ROM `.mem` files to initialize memory tables for all $196 \times 32 + 32 \times 10 = 6592$ edge functions.

#### [tb_mnist_kan.sv](file:///home/phil/devel/FPGA/KAN_LUT/examples/MNIST/FPGA/tb_mnist_kan.sv)
A SystemVerilog testbench that:
* Feeds test digits into `mnist_kan_top.sv`.
* Verifies clock-cycle by clock-cycle that outputs match the outputs generated by `examples/MNIST/inference.jl`.

### Paradigm 2: Parameterized Generic KAN IP Core

We will create a top-level `hardware/` directory for the generic core.

#### [kan_bram_bank.sv](file:///home/phil/devel/FPGA/KAN_LUT/hardware/kan_bram_bank.sv)
A parameterized memory bank representing the L-LUT tables.
* Supports dual-port reads for time-multiplexed lookups.
* Includes a write port to load/update weights at boot time.

#### [kan_generic_core.sv](file:///home/phil/devel/FPGA/KAN_LUT/hardware/kan_generic_core.sv)
The parameterized computational core:
* **Parameters**: `INPUT_DIM`, `OUTPUT_DIM`, `LUT_DEPTH` (256), `DATA_WIDTH` (8), `FRACTIONAL_BITS` (4), `PARALLELISM` (number of parallel lookups per clock cycle, e.g. 4, 8, or 16).
* **Datapath**: A time-multiplexed set of parallel lookup units feeding a parameterized adder tree.
* **Control**: An FSM that sequences through inputs in chunks of size `PARALLELISM`, accumulates the sums, applies division/saturation, and stores the intermediate outputs in a local buffer.

#### [tb_generic_core.sv](file:///home/phil/devel/FPGA/KAN_LUT/hardware/tb_generic_core.sv)
A SystemVerilog testbench validating the generic core on both the Moons ($2 \rightarrow 8 \rightarrow 2$) and MNIST ($196 \rightarrow 32 \rightarrow 10$) models.

## Detailed Design Specifications

### 1. Balanced Pipelined Adder Tree (Paradigm 1)
For $D$ inputs, a binary adder tree requires $\lceil \log_2(D) \rceil$ stages:
* **MNIST Layer 1**: $D = 196$, requires 8 pipeline stages.
* **MNIST Layer 2**: $D = 32$, requires 5 pipeline stages.

To prevent overflow during summation, the width of the adders at stage $s$ will be $W + s$ bits (starting from 8-bit outputs of the ROMs).
At the final stage of the tree, we perform:
1. Division: Rounding division by $2^k = 16$ (adding $2^{k-1} = 8$ and right shifting by 4).
2. Saturation: Clamping the value to range `[0, 255]` using a SystemVerilog ternary operator:
   ```systemverilog
   assign out = (div_val > 255) ? 8'hFF : (div_val < 0) ? 8'h00 : div_val[7:0];
   ```

### 2. Time-Multiplexed Accumulation (Paradigm 2)
For an input vector of dimension `INPUT_DIM` and `PARALLELISM = P`:
* The core takes $\lceil \text{INPUT_DIM}/P \rceil$ clock cycles to process all inputs for a single output channel.
* For $D_{out}$ output channels, it loops sequentially, requiring $D_{out} \times \lceil \text{INPUT_DIM}/P \rceil$ clock cycles per layer.
* High flexibility: Only requires $P$ physical adders instead of $D_{in} \times D_{out}$ adders.

## Verification & Comparison Plan

We will simulate both designs using **Icarus Verilog** (or Verilator if available) to compare:
1. **Throughput / Latency**: Cycle count per inference.
2. **Resource Consumption**: Estimating Logic Elements (LEs) and Block RAMs (BRAMs).
3. **Synthesis Flexibility**: Evaluating the trade-offs of dynamic reloading vs. custom compilation.
